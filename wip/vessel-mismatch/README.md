# Vessel-mismatch action lists

Single-source-of-truth policy: `data/vessel-list.csv` is canonical. Every
vessel-like mention surfaced by `mtss validate ingest` (check #36) is either:

1. **Typo** — fix it (map detected → canonical, retag chunks), **or**
2. **Untracked vessel** — park the raw mention on
   `chunks.metadata.unknown_vessel_names` so a future register update can
   pick it up cheaply, **or**
3. **Extractor noise** — not a real vessel mention; feed into the regex
   blocklist (task #20). No DB action.

This folder holds the curated lists for the next apply pass plus a
reproducible generator and an env-aware playbook.

## Files

| File | Purpose | Consumed by |
|---|---|---|
| `typo_mapping.csv` | `detected_name,vessel_name,occurrences` mapping | `scripts/apply_vessel_name_mapping.py --mapping ...` |
| `untracked_parking.txt` | Raw mentions to stamp on `unknown_vessel_names` | (see "Park untracked" below) |
| `extractor_noise.txt` | False-positive mentions to suppress at extraction time | Manual review → `src/mtss/processing/vessel_mention_extractor.py` `HARDCODED_NOISE` |
| `mismatch_report.md` | Human-readable categorisation report | Reading only |
| `generate.py` | Reproducible script that produces the three lists above | `uv run python wip/vessel-mismatch/generate.py` |

## Re-generating the lists on another workstation

The lists are derived from a local `data/ingest.db`. To refresh:

```bash
cd <repo-root>
OPENROUTER_API_KEY=test-key uv run python wip/vessel-mismatch/generate.py
```

`OPENROUTER_API_KEY` only needs to be set so `Settings()` constructs; the
generator does not call any LLM. The script is read-only against
`data/ingest.db` and `data/vessel-list.csv`.

After generating, eyeball the three files and edit `typo_mapping.csv`
manually if any auto-suggested mapping looks wrong before applying.

## Apply playbook (test → prod)

The other workstation has both `.env.test` and `.env` files. **Always apply
to test first**, validate, then promote to prod. Snapshot before each step
so a rollback is one command away.

### Pre-flight (any env)

```bash
# Confirm tooling installed
uv sync --extra ingest

# Snapshot the canonical vessel list before
uv run python scripts/snapshot_vessel_uuids.py > /tmp/vessels_before.json
```

### TEST env

```bash
set -a; source .env.test; set +a

# 1. Apply typo mapping — DRY RUN first
uv run python scripts/apply_vessel_name_mapping.py \
    --mapping wip/vessel-mismatch/typo_mapping.csv \
    --dry-run

# Review the planned changes, then commit:
uv run python scripts/apply_vessel_name_mapping.py \
    --mapping wip/vessel-mismatch/typo_mapping.csv \
    --apply

# 2. Park untracked vessels — DRY RUN first
uv run python scripts/annotate_unknown_vessel_mentions.py --dry-run
uv run python scripts/annotate_unknown_vessel_mentions.py --apply

# 3. Re-tag chunks against the now-cleaner register
uv run python scripts/retag_vessel_ids_from_db.py --dry-run
uv run python scripts/retag_vessel_ids_from_db.py --apply

# 4. Validate — check #36 unknown count should drop substantially
uv run mtss validate ingest

# 5. Snapshot after for diff
uv run python scripts/snapshot_vessel_uuids.py > /tmp/vessels_after_test.json
diff /tmp/vessels_before.json /tmp/vessels_after_test.json
```

### PROD env

Only after the test-env diff looks right.

```bash
set -a; source .env; set +a

# Backup before mutating prod (writes to data/backup/<timestamp>/)
uv run mtss backup

# Snapshot
uv run python scripts/snapshot_vessel_uuids.py > /tmp/vessels_before_prod.json

# Repeat steps 1–4 from above against prod
uv run python scripts/apply_vessel_name_mapping.py \
    --mapping wip/vessel-mismatch/typo_mapping.csv \
    --apply
uv run python scripts/annotate_unknown_vessel_mentions.py --apply
uv run python scripts/retag_vessel_ids_from_db.py --apply
uv run mtss validate ingest

uv run python scripts/snapshot_vessel_uuids.py > /tmp/vessels_after_prod.json
diff /tmp/vessels_before_prod.json /tmp/vessels_after_prod.json
```

## Caveats

- `scripts/annotate_unknown_vessel_mentions.py` does its own extraction
  with the same regex as `vessel_mention_extractor.py`, so its parked set
  may not match `untracked_parking.txt` 1:1 — it skips entries that
  resolve via aliases after the typo pass, which is correct behaviour.
- The typo mapping was generated by `difflib.get_close_matches` with
  cutoff 0.85. **A few entries are wrong** — e.g. `MARAN POSEIDON AND` →
  `MARAN POSEIDON` is right, but `MARAN THE` → `MARAN THETIS` is a guess.
  Hand-review before `--apply`.
- `untracked_parking.txt` includes entries like
  `ANTONIS L. ANGELICOUSSIS` which is a person, not a vessel — these
  will be parked in `unknown_vessel_names` harmlessly but should
  ideally be blocklisted (#20).

## Rollback

If the apply produces unexpected results:

- `data/backup/<timestamp>/` (created by `mtss backup`) carries a
  full archive snapshot.
- `vessels` table snapshots at `/tmp/vessels_before*.json` are the
  authoritative pre-state for the canonical register.
- For chunk metadata, `chunks` table changes are constrained to the
  `metadata.vessel_ids` / `vessel_types` / `vessel_classes` /
  `unknown_vessel_names` keys. A re-ingest from the source EMLs
  rebuilds these from scratch via the matcher.

## How the lists were generated

See `generate.py` for the exact code, but the high level:

1. Iterate every `documents.title|source_title|file_name` and every
   `chunks.content` in `data/ingest.db`.
2. Apply the regex extractor in
   `src/mtss/processing/vessel_mention_extractor.py` to surface
   vessel-like name candidates.
3. Drop any candidate that exactly matches a canonical name or alias
   from `data/vessel-list.csv`.
4. Categorise the remainder:
   - **noise** — well-known false-positive patterns (`MARAN <org>`,
     business phrases caught by `M.[TV]`, canonical-vessel + adjacent
     word concatenations).
   - **typo** — within edit distance ≤ 0.85 of a canonical name (via
     `difflib.get_close_matches`), and matches the `MARAN X` /
     `X VOYAGER` shape.
   - **untracked** — everything else.

The validate-time check (`mtss validate ingest` check #36) emits the same
candidate set as warnings; this generator adds the categorisation pass on
top.
